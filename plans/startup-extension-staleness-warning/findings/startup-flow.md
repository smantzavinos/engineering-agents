# Pi startup and extension wrapper flow

**Created:** 2026-05-19
**Topic:** Current Pi launch/configuration flow, managed extension wrapping, and the earliest observable safe point for a non-blocking startup warning
**Plan:** `plans/startup-extension-staleness-warning/brief.md`

## Summary
This repo does not currently ship a custom runtime launcher for `pi`. The Nix flake exposes a Home Manager Pi module, that module installs the upstream Pi binary from `llm-agents.nix`, and the README tells users to start Pi by invoking `pi` directly (`flake.nix:62-76`, `nix/modules/pi/default.nix:345-349`, `README.md:41-56`, `README.md:145-158`). Repo-owned startup-related work today happens in Home Manager activation, not on each `pi` process start.

The extension "wrapper" mentioned in the brief is implemented as generated facade packages under `~/.pi/agent/packages/<packageId>`. The Pi settings generated by the module point Pi at those local facade directories instead of the original installed package roots (`nix/modules/pi/default.nix:237-270`, `nix/modules/pi/default.nix:557-658`). The compiler rewrites exposed resources to facade-local `./_source/...` paths and writes a synthetic `package.json` with `version: "0.0.0-generated"`, while the original source type/spec is preserved only in `meta/source.json` (`nix/modules/pi/compile-managed-packages.mjs:539-577`, `nix/modules/pi/compile-managed-packages.mjs:610-646`). In the observed runtime sequence used by the proof-set harness, Pi loads settings first, resolves configured packages next, and then loads resources; that means the earliest observable point where both configured package paths and original source provenance are available is immediately after configured packages are resolved and before/while `DefaultResourceLoader.reload()` runs (`tests/scripts/resource-snapshot.mjs:527-595`).

## Findings

### Pi is installed directly; the repo does not currently wrap the `pi` executable
- The flake imports the Pi Home Manager module from `./nix/modules/pi` and exposes it as `homeManagerModules.pi`; no wrapper package for `pi` is defined there (`flake.nix:62-76`).
- When the Pi module is enabled, `home.packages` includes `piPkg = llmAgents.packages.${pkgs.system}.pi`, which is the upstream Pi package, plus auxiliary tools like `ast-grep` and optional `gitnexus` (`nix/modules/pi/default.nix:230`, `nix/modules/pi/default.nix:345-349`).
- The repository README instructs users to authenticate and start sessions by running `pi` directly, with no alternate command shown (`README.md:47-56`, `README.md:145-158`).
- The only packaged helper binary added by this repo is `check-updates`, exposed through the flake overlay; there is no analogous packaged `pi` launcher in `flake.nix` (`flake.nix:81-86`).

### Pi configuration is generated into `~/.pi/agent/settings.json`, and managed packages are configured as local facade paths
- The Nix module builds a `piSettings` JSON object and writes it to a generated file, then the `installPiSettings` activation step copies or merges that content into `$HOME/.pi/agent/settings.json` (`nix/modules/pi/default.nix:237-275`, `nix/modules/pi/default.nix:557-573`).
- In that generated settings object, `packages` is set to `./packages/${packageId}` for every managed non-local package (`nix/modules/pi/default.nix:263-269`). This is the key indirection: Pi is configured to load local package directories under the agent dir, not the original npm/git installation roots.
- The checked-in repo-local `.pi/settings.json` only contains model/subagent overrides and an empty `packages` array (`.pi/settings.json:1-26`). Based on the module code above, runtime package configuration for installed users comes from `~/.pi/agent/settings.json`, not from the repo-local settings file.
- The proof-set harness confirms this runtime shape: configured packages appear as `./packages/<packageId>` and resolve to installed paths under `/fixture/.pi/agent/packages/<packageId>` (`tests/spec-fixtures/resource-snapshot.v2.ok.json:19-54`).

### Managed package installation has two layers: materialized source packages and generated facade packages
- During `installPiExtensions`, npm-backed packages are installed globally into the user prefix only when the requested version is not already present (`nix/modules/pi/default.nix:594-610`).
- Git-backed packages are always reinstalled into the same npm prefix using `npm install -g --install-links --legacy-peer-deps`, after uninstalling the package name and deleting its previous materialized directory (`nix/modules/pi/default.nix:612-624`).
- After installation, the activation script writes `managed-packages.declarations.json` where each non-local package records the original `source.type`, `source.spec`, and a `materializedPath` pointing at `$HOME/.pi/packages/lib/node_modules/<packageName>` (`nix/modules/pi/default.nix:626-642`).
- The compiler then writes outputs into two runtime directories: `~/.pi/agent/sources/<materializedKey>` for shared source roots and `~/.pi/agent/packages/<packageId>` for the generated facades (`nix/modules/pi/compile-managed-packages.mjs:649-664`, `nix/modules/pi/compile-managed-packages.mjs:725-748`, `nix/modules/pi/compile-managed-packages.mjs:762-790`).
- Shared-source behavior is explicit: the compiler hashes `source.type + source.spec` into `materializedKey`, caches each materialized source once, and lets multiple facades point at the same source root (`nix/modules/pi/compile-managed-packages.mjs:125-140`, `nix/modules/pi/compile-managed-packages.mjs:725-739`). The snapshot fixture shows this for `pi-ext-leader-key` and `pi-ext-review`, which share the same `sourceRoot` and `materializedKey` (`tests/spec-fixtures/resource-snapshot.v2.ok.json:208-229`, `tests/spec-fixtures/resource-snapshot.v2.ok.json:252-275`).

### The wrapper works by replacing original package identity with a synthetic facade package
- `toFacadePaths()` rewrites selected extensions, prompts, and themes to `./_source/...` paths inside the facade package, and rewrites root-level skills through a facade-local `skills/<name>/SKILL.md` directory so Pi accepts them (`nix/modules/pi/compile-managed-packages.mjs:539-558`, `nix/modules/pi/compile-managed-packages.mjs:620-633`).
- `writeFacadePackage()` generates a facade `package.json` with:
  - `name: pkg.packageId`
  - `private: true`
  - `version: '0.0.0-generated'`
  - `pi: facadePaths`
  (`nix/modules/pi/compile-managed-packages.mjs:636-644`)
- The original source details are not kept in that generated `package.json`; instead they are written to `meta/source.json` via `buildMetadata()` (`nix/modules/pi/compile-managed-packages.mjs:560-577`, `nix/modules/pi/compile-managed-packages.mjs:643-644`).
- The proof-set contract asserts this provenance sidecar is required and must preserve the original `source.spec` and `sourceManifestName` (`tests/scripts/assert-contract.sh:137-156`).
- The snapshot fixture demonstrates the split clearly: `configuredPackagePath` is `./packages/pi-subagents`, the facade path is `/fixture/.pi/agent/packages/pi-subagents`, but the original git spec is only present inside `sourceProvenance.source.spec` (`tests/spec-fixtures/resource-snapshot.v2.ok.json:105-133`).

### The manual update workflow only checks npm package versions; git sources are reported but not evaluated for staleness
- The repository’s update helper is `scripts/check-updates.sh`, and the flake overlay packages it as `check-updates` (`scripts/check-updates.sh:1-9`, `flake.nix:81-86`).
- That script parses the `piPackages` declaration block from `nix/modules/pi/default.nix`, validates the declaration contract, and emits a normalized package list (`scripts/check-updates.sh:57-191`).
- For `npm` sources, it calls `npm view <packageName> version` and compares the declared version with the latest registry version (`scripts/check-updates.sh:391-400`).
- For `git` sources, it does not compute freshness. It only prints `[PI_PACKAGE_WARN_GIT_SOURCE_MANUAL_UPDATE] <package_id> uses git source <source_spec>` to stderr (`scripts/check-updates.sh:402-404`).
- This aligns with the brief’s statement that the manual checker exists but does not restore git-based stale detection; the codebase evidence is that git packages are surfaced only as a generic manual-update warning, not as stale/not-stale results.

### The closest observable runtime startup sequence is: locate Pi -> reload settings -> resolve configured packages -> load resources
- The proof-set snapshot helper imports Pi’s runtime module from the installed `pi` binary path, then constructs `SettingsManager`, `DefaultPackageManager`, and `DefaultResourceLoader` from Pi’s own code (`tests/scripts/resource-snapshot.mjs:189-214`, `tests/scripts/resource-snapshot.mjs:532-557`).
- The call order in that helper is:
  1. `SettingsManager.create(cwd, agentDir)`
  2. `await settingsManager.reload()`
  3. construct `DefaultPackageManager`
  4. construct `DefaultResourceLoader`
  5. `await resourceLoader.reload()`
  6. inspect `packageManager.listConfiguredPackages()` and loaded resources
  (`tests/scripts/resource-snapshot.mjs:539-565`)
- For each configured package, the helper reads the facade manifest and `meta/source.json` from `installedPath`, which is how it reconstructs original package/source provenance (`tests/scripts/resource-snapshot.mjs:233-244`, `tests/scripts/resource-snapshot.mjs:565-580`).
- The test README explicitly describes this script as “Live Pi state snapshot using Pi’s `DefaultResourceLoader`,” which is the repo’s best direct evidence of Pi’s resource-loading flow (`tests/README.md:35-41`).

### Earliest observable safe point for a non-blocking startup warning
- There is no repo-owned production code in this repository that currently runs on every `pi` process start. The repo-owned automation runs at Home Manager activation (`installPiSettings`, `installPiExtensions`, optional `installVisualExplainer`, optional `installAgentKit`) rather than at CLI startup (`nix/modules/pi/default.nix:553-713`).
- Because of that, there is no existing repo-local pre-launch hook before `pi` starts. The current user-facing startup path is still “invoke upstream `pi` directly” (`README.md:49-56`, `README.md:145-158`).
- Within the observable Pi runtime sequence, the earliest point where enough information exists to classify repo-managed packages is **after settings reload and configured-package resolution, but before/while resources are loaded**. At that point the system already knows:
  - which packages are configured (`packageManager.listConfiguredPackages()`), and
  - where each facade lives (`installedPath`), and
  - how to recover original source type/spec from `meta/source.json`
  (`tests/scripts/resource-snapshot.mjs:565-580`).
- This is the earliest point evidenced in-repo where a startup warning could distinguish facade package IDs from underlying source provenance without needing to infer anything from the synthetic facade `package.json`.
- Earlier than that, only the static declaration source in `nix/modules/pi/default.nix` is available; later than that, `DefaultResourceLoader.reload()` has already begun loading wrapped resources (`tests/scripts/resource-snapshot.mjs:553-563`).

## Constraints Discovered
- The repo currently replaces original package manifests with generated facade manifests that always use `version: "0.0.0-generated"`; any freshness logic that only inspects the facade manifest will lose the original version/spec (`nix/modules/pi/compile-managed-packages.mjs:636-644`).
- Original source identity is preserved only in `meta/source.json`, so any runtime stale-check logic needs access to the facade directory and that sidecar metadata (`nix/modules/pi/compile-managed-packages.mjs:560-577`, `tests/scripts/assert-contract.sh:137-156`).
- Git-backed managed packages can share one materialized source root across multiple facade package IDs; stale status for one source may need to fan out to multiple package IDs (`nix/modules/pi/compile-managed-packages.mjs:725-739`, `tests/spec-fixtures/resource-snapshot.v2.ok.json:208-275`).
- The existing repo-owned update workflow has authoritative npm-version semantics but no confirmed stale/not-stale result for git sources (`scripts/check-updates.sh:391-404`).
- Existing compile-time warnings are produced at activation and written to `managed-packages.report.json`; the repo currently surfaces those in tests, not at interactive Pi startup (`nix/modules/pi/default.nix:644-646`, `tests/scripts/resource-snapshot.mjs:476-494`).

## Risks
- If a startup warning is attached only to facade package metadata, it may report misleading version information because the facade manifest is synthetic (`nix/modules/pi/compile-managed-packages.mjs:636-644`).
- Shared-source facades such as `pi-ext-leader-key` and `pi-ext-review` could cause duplicated warnings unless source-level results are deduplicated before being mapped back to package IDs (`tests/spec-fixtures/resource-snapshot.v2.ok.json:208-275`).
- The repo has no current production startup hook of its own, so any startup-time warning behavior will need to be introduced either before `pi` launch or at a Pi runtime integration point not presently implemented in this repository; there is no existing repo file that already executes at every launch (`flake.nix:62-76`, `nix/modules/pi/default.nix:345-349`, `nix/modules/pi/default.nix:553-713`).
- It is unclear from this repo alone whether Pi’s extension lifecycle exposes a built-in non-blocking “startup notification” callback before interactive use; the repo only demonstrates resource loading through Pi runtime classes in test code, not a production extension implementation that emits such a warning (`tests/scripts/resource-snapshot.mjs:532-559`).
